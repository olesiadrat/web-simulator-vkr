# ВКР — веб-тренажёр по тестированию пользовательских интерфейсов

Выпускная квалификационная работа, посвящённая разработке веб-приложения для тренировки практических навыков ручного тестирования UI.

Пользователь проходит интерактивный сценарий, анализирует интерфейс, фиксирует найденные дефекты в баг-панели и получает результат сессии.

---

## 1. Назначение системы

Цель проекта — предоставить учебную среду, в которой можно:

- работать с интерфейсом, сгенерированным из JSON-сценария;
- находить UI/UX-дефекты;
- создавать баг-репорты по шаблону;
- просматривать накопленные баги в рамках текущей сессии.

Система ориентирована на MVP-подход: ключевой акцент сделан на архитектуре, интерактивности интерфейса и базовом процессе фиксации дефектов.

---

## 2. Архитектура

Проект реализован в клиент-серверной архитектуре:

- **Frontend:** React + TypeScript + Vite
- **Backend:** Django + Django REST Framework
- **Database:** PostgreSQL (для локальной разработки допускается SQLite)
- **Взаимодействие:** REST API

## 4. Основной функционал (реализовано в MVP)

### 4.1 Работа со сценариями

- хранение сценариев в БД (JSON-структура);
- получение списка сценариев через API;
- запуск пользовательской сессии по выбранному сценарию.

### 4.2 Динамический рендеринг интерфейса

- отрисовка UI на frontend на основе JSON;
- поддержка интерактивных элементов: текст, поля ввода, селекты, даты, кнопки, карточки, изображения, галерея;
- переходы между экранами сценария;
- возможность выбирать элементы интерфейса для оформления баг-репорта.

### 4.3 Панель баг-репорта

- форма с полями:
  - описание ошибки;
  - шаги воспроизведения (динамический список шагов);
  - expected behavior;
  - actual behavior;
  - элемент интерфейса;
- клиентская валидация базовых обязательных полей;
- предпросмотр бага в модальном окне перед отправкой;
- просмотр и редактирование сохранённых багов в modal;
- заглушка «Проверить баг».

### 4.4 Результаты сессии

- список найденных багов в рамках текущей сессии (на текущем этапе хранится локально на frontend);
- отображение количества сохранённых баг-репортов;
- завершение сессии через backend API.

### 4.5 Учебные требования к элементам UI

В сценарий встроены тултипы с формальными требованиями к поведению элементов (валидация, ограничения ввода, ожидаемая логика кнопок и расчётов). Часть требований намеренно не соблюдается — это используется как учебный материал для поиска дефектов.

## 5. Модель данных (backend)

Реализованы сущности:

- **Scenario**
  - `id`
  - `title`
  - `description`
  - `json_structure`
  - `created_at`

- **Session**
  - `id`
  - `scenario`
  - `start_time`
  - `end_time`

- **BugReport**
  - `id`
  - `session`
  - `description`
  - `expected`
  - `actual`
  - `ui_element`
  - `created_at`

- **ReferenceBug**
  - `id`
  - `scenario`
  - `description`
  - `expected_behavior`
  - `ui_element`

---

## 6. REST API

Базовые endpoints:

- `GET /api/scenarios/` — список сценариев
- `POST /api/sessions/` — старт сессии
- `PATCH /api/sessions/{id}/finish/` — завершение сессии
- `GET /api/bugs/?session_id={id}` — список багов сессии
- `POST /api/bugs/` — создание баг-репорта
- `GET /api/reference-bugs/?scenario_id={id}` — эталонные баги сценария

## 8. Запуск проекта локально

### 8.1 Backend

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python manage.py migrate
python manage.py seed_demo
python manage.py runserver
```

Backend/API по умолчанию:

```text
http://127.0.0.1:8000/api/
```

Если PostgreSQL пока не настроен, в `backend/.env` можно временно включить:

```text
USE_SQLITE=True
```

Для итоговой версии ВКР рекомендуется использовать PostgreSQL (`USE_SQLITE=False`).

### 8.2 Frontend

```bash
cd frontend
npm install --cache /tmp/web-simulator-npm-cache
npm run dev
```

Frontend по умолчанию:

```text
http://127.0.0.1:5173/
```

Если `node` установлен в `/usr/local/bin`, но не добавлен в `PATH`:

```bash
PATH=/usr/local/bin:$PATH npm run dev
```

---

## 9. Ограничения текущего этапа

- На текущем этапе список багов в баг-панели хранится локально в состоянии frontend (без финальной интеграции сохранения через API из формы панели).
- Не реализованы NLP/автопроверка качества текста баг-репорта.
- Реализован MVP, ориентированный на демонстрацию архитектуры, UI-сценария и процесса фиксации дефектов.
